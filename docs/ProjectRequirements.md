# Campus Bulletin Board System 项目要求


## 一、项目概述

**项目名称**：Campus Bulletin Board System (校园论坛)

**项目描述**：一个面向大学校园的在线论坛系统，提供用户注册、发帖、评论、点赞等功能

## 二、大程要求

组得分 = 教师打分 * 50% +（各组长打分和）/ 组数 * 50%

各组长打分在Review时进行

教师打分包括两部分：汇报分（主要依据汇报效果以及提交物质量）+ 过程分（主要依据Git评估开发过程的合理性，如开发流程是否规范、AI使用是否规范） 

组员得分 = 组员贡献率 * 该项组得分 * 组人数

组员总得分 = 各项组员得分 * 各项占比

需求分析30%、项目计划10%、系统设计20%、原型系统20%、系统测试20%


## 三、子系统拆分

### 3.1 用户与认证子系统

**职责**：完成用户身份管理与访问控制。  
**核心功能**：
- 用户注册、登录、登出
- 密码加密存储与重置
- JWT 鉴权与会话管理（可结合 Redis）
- 用户角色与权限控制（普通用户/管理员）

### 3.2 帖子与分区子系统

**职责**：管理论坛分区与主题帖全生命周期。  
**核心功能**：
- 分区（板块）创建、编辑、删除、展示
- 发帖、编辑、删除、置顶、加精
- 帖子详情与列表查询（分页、排序）

### 3.3 评论与互动子系统

**职责**：承载帖子下的互动行为。  
**核心功能**：
- 评论发布、回复、删除
- 点赞/取消点赞（帖子、评论）
- 互动状态聚合（点赞数、评论数）
  
### 3.4 通知与消息子系统

**职责**：向用户传递关键事件。  
**核心功能**：
- 评论通知、回复通知、点赞通知
- 系统公告与站内信
- 未读计数与已读状态管理

### 3.5 搜索与推荐子系统

**职责**：提升内容发现效率。  
**核心功能**：
- 帖子标题/正文关键词搜索
- 热门帖子排序（按热度、时间）
- 个性化推荐（可作为迭代目标）


### 3.6 管理后台子系统

**职责**：提供平台运营与维护入口。  
**核心功能**：
- 用户管理（封禁、解封、权限调整）
- 板块管理、公告管理
- 举报处理面板与审核统计
- 操作审计与日志查看 
 
## 四、功能优先级与迭代规划
 
| 优先级 | 子系统 | 说明 |
| --- | --- | --- |
| MVP 必做 | 3.1、3.2、3.3 | 用户认证、帖子分区、评论互动 |
| MVP 必做 | 3.4（基础） | 评论/回复/点赞通知、未读计数 |
| MVP 必做 | 3.6（基础） | 用户管理、板块管理 |
| 迭代一 | 3.4（完整） | 系统公告与站内信 |
| 迭代一 | 3.6（完整） | 公告管理、举报处理 |
| 迭代二 | 3.5 | 关键词搜索、热门排序 |
| 迭代三 | 3.5（进阶） | 个性化推荐 |
| 迭代三 | 3.6（进阶） | 操作审计与日志 |


## 五、组内分工

### 5.1 组长

- 搭建开发环境，初始并管理 Git 仓库
- 负责定义数据库表结构
- 负责分支合并时的代码审核
- 负责项目文档的撰写与维护，包括 API 文档
- 参与后端的工作

### 5.2 前端（两人）

**前端A**面向用户端，负责实现用户注册、登录、发帖、评论、通知等核心功能的界面与交互。

**前端B**面向管理端，负责实现用户管理、板块管理、公告管理、举报处理等后台功能的界面与交互。

### 5.3 后端（两人）

**后端A**负责系统基础层，涉及子系统 3.1（用户认证）+ 3.2（帖子分区）

**后端B**负责互动与管理，涉及子系统 3.3（评论互动）+ 3.4（通知）+ 3.6（管理后台）

> 说明：管理后台的本质是对评论、通知、用户数据的增删改查接口，后端B做评论互动后顺手承接对应管理接口更自然连贯。搜索推荐（3.5）作为后期迭代目标，由全组共同推进。

### 5.4 测试兼文档（一人）

- 前期（需求分析与设计阶段）：侧重文档撰写与维护
- 中期（开发阶段）：编写单元测试、集成测试，配置 CI/CD
- 后期（测试阶段）：全面回归测试与缺陷跟踪
